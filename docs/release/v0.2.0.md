# pretty-lang v0.2.0 — The Core

**The formatter engine lands.** v0.1.0 was a scaffold with no domain logic; v0.2.0 makes pretty-lang a working, language-agnostic pretty-printer. It introduces the `Doc` layout algebra and a linear-time renderer that reflows any syntax tree to a target width — the reusable half of a `gofmt`-style formatter for any language.

## What is pretty-lang?

A pretty-printer that turns a syntax tree into laid-out source text. It knows nothing about grammars: you describe a layout with a handful of combinators — group these pieces, break here, indent that — and pretty-lang decides where the lines actually break for a given width. The same document renders densely at width 100 and stacked at width 20, with no branching in your code. It is `no_std` (needs only `alloc`) and contains no `unsafe` (`#![forbid(unsafe_code)]`).

## What's new in 0.2.0

### `Doc` — the layout algebra

The whole public surface is one type, [`Doc`], with a small, complete set of operations. You build a document from any AST and render it:

```rust
use pretty_lang::Doc;

let call = Doc::text("f(")
    .append(
        Doc::softline()
            .append(Doc::join(
                Doc::text(",").append(Doc::line()),
                ["a", "b", "c"].map(Doc::text),
            ))
            .nest(4),
    )
    .append(Doc::softline())
    .append(Doc::text(")"))
    .group();

assert_eq!(call.render(80), "f(a, b, c)");
assert_eq!(call.render(6), "f(\n    a,\n    b,\n    c\n)");
```

The combinators:

- **Build:** `Doc::text` (literal, unbreakable), `Doc::nil` (empty), `Doc::concat` (fold), `Doc::join` (intersperse a separator).
- **Break:** `Doc::line` (space when flat, newline when broken), `Doc::softline` (nothing when flat), `Doc::hardline` (always a newline; forces enclosing groups to break).
- **Combine:** `Doc::append` (concatenate), `Doc::nest` (indent inner breaks), `Doc::group` (lay flat if it fits, otherwise break all-or-nothing).

That is enough to format any language. A JSON pretty-printer is a single walk over the value tree — see [`examples/json.rs`](../../examples/json.rs), which renders dense one-liners where they fit and indented blocks where they do not, with no width logic in the walk.

### A linear-time renderer

The engine is Wadler's *A Prettier Printer* in Lindig's imperative form. Rendering is one pass over the document with an explicit work stack; at each `group` a width-bounded look-ahead answers "does the flat form fit the rest of this line?" and stops the instant the answer is known — at the first newline or the first column past the width. Total work is proportional to document size, not quadratic.

Three ways to render:

- `Doc::render(width) -> String` — the common case.
- `Doc::render_into(width, &mut impl core::fmt::Write)` — stream into a caller-owned buffer, no intermediate `String`.
- `Doc::render_writer(width, &mut impl std::io::Write)` — stream to a file, socket, or stdout, propagating the first I/O error (behind the default `std` feature).

### No recursion, anywhere

Both the render pass and the look-ahead run on heap work stacks, not the call stack, so a document nested tens of thousands of levels deep — a long chain of binary expressions, say — renders without overflowing. The `Drop` impl dismantles such a document the same way: a uniquely-owned deep spine is torn down iteratively, while leaves and shared nodes take a branch-only fast path that allocates nothing. A deeply nested document neither renders nor drops recursively.

### Property tests

`tests/proptests.rs` exercises the invariants that must hold for every document across randomized inputs:

- **Flat-layout oracle** — a fully grouped, hardline-free document at a wide width matches a flat string built independently from the leaves, so the check does not just re-run the code under test.
- **Panic safety** — rendering never panics at any width from 0 upward, including documents full of forced breaks.
- **`render_into` consistency** — it produces exactly what `render` returns.
- **Associativity** — `a.append(b).append(c)` and `a.append(b.append(c))` render identically.
- **Idempotence** — `d.group().group()` renders the same as `d.group()`.

### Benchmarks

`benches/bench.rs` measures two representative workloads — a JSON-like tree and a wide function call — both where they fit flat and where they are forced to break, plus build-and-render. Local Criterion means (Linux x86_64 / WSL2, Rust stable, release build):

| Workload | Layout | Time |
|----------|--------|-----:|
| JSON tree (~85 nodes, ~340 B out) | flat | ~1.8 µs (**~550 MiB/s**) |
| JSON tree (~85 nodes) | broken | ~4.0 µs |
| Call, 128 arguments | flat | ~1.9 µs |
| Call, 128 arguments | broken | ~2.7 µs |
| Call, 8 arguments | flat | ~190 ns |

Numbers vary by CPU and environment; run the suite on your target to establish a baseline.

### Documentation

`docs/API.md` is a full reference: every public item with its parameters, return type, contract, and at least two runnable examples, plus a recipes section and design notes. Three runnable examples ship in `examples/` — `quick_start`, `json`, and `rust_signature`.

## Breaking changes

None from a consumer's perspective — v0.1.0 exposed no API. One scaffold cleanup worth noting: the unused `serde` feature and dependency were removed, since a layout document has no serialization use case. The only feature is now `std` (default), which gates the `io::Write` renderer.
